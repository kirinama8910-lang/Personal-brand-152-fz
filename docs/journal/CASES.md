# CASES.md — журнал проблемных кейсов

> **Зачем:** фиксировать ситуации, когда система работает не так, как ожидалось. Каждый кейс — будущий тест и основание для задачи.

---

## Когда фиксировать кейс

- Результат работы системы не совпадает с ожиданием (баг в квизе, форме, юр.документах, сценарии бота).
- Исправлен баг — запиши со статусом «исправлено», чтобы при регрессии быстро вспомнить.
- Пользователь сообщил о проблеме.

## Когда Claude должен предлагать запись

После любого из событий ниже Claude формулирует короткий вопрос: «Зафиксировать в CASES.md? [да / нет / правка]».

- Поймал баг и починил его (даже мелкий).
- Воспроизвёл странное поведение, но причина пока неясна.
- Пользователь описал баг, который не удалось воспроизвести.

## Как заполнить за 2 минуты

1. Скопировать шаблон ниже, вставить в конец раздела «Кейсы».
2. Заполнить поля со звёздочкой ★ — остальное можно дописать позже.
3. Номер — следующий по порядку (CASE-001, CASE-002 …).

## Шаблон

```
### CASE-NNN: [суть проблемы в 5–7 словах]
- ★ **Дата:** ГГГГ-ММ-ДД
- ★ **Вход:** [что подали на вход — URL, файл, действие, последовательность кликов]
- ★ **Что произошло:** [что система выдала]
- ★ **Что ожидалось:** [что должно было быть]
- **Подробности:** [дополнительный контекст]
- **Причина:** [если понятна, иначе не заполняй]
- **Статус:** открыт | исправлено в [задача / коммит] | не воспроизводится | won't fix (причина)
```
