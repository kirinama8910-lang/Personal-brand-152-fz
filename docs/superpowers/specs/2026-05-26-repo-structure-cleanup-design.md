# Уборка структуры репозитория и онбординг Claude Code

**Дата:** 2026-05-26
**Тип:** реорганизация репозитория + расширение CLAUDE.md
**Цель:** очистить корень от мусора, перенести скиллы в `.claude/skills/`, нарезать тяжёлый SPEC.md, расширить CLAUDE.md для эффективного cold-start новых чатов и снижения потребления токенов.

---

## 1. Проблема

### Что сейчас плохо

1. **Корень засорён.** В корне 9 `.md`-файлов вперемешку: актуальные (`README.md`, `CLAUDE.md`, `PROJECT_PASSPORT.md`, `SPEC.md`), активные журналы (`CASES.md`, `PATTERNS.md`, `DECISIONS.md`), частичный дубль глобальных правил (`CLAUDE_CODE_RULES.md`), плюс файлы скиллов не в своём доме (`SKILL.md`, `uluchshatel-promptov.skill`).
2. **SPEC.md — 60 KB одним полотном.** При cold-start новых чатов Claude нередко затягивает его целиком и тратит токены впустую.
3. **CLAUDE.md фокусируется только на запретах языка.** Нет карты проекта, нет инструкции «когда что читать», нет упоминания паспорта, журналов и скиллов. Из-за этого Claude в новом чате идёт «вслепую» и часто читает лишнее.
4. **Скиллы не в `.claude/skills/`.** `SKILL.md` (frontend-design 152-ФЗ) лежит в корне. `uluchshatel-promptov` — zip-архив в корне, не распакован, скилл не активен.
5. **Журналы (`CASES`, `PATTERNS`, `DECISIONS`) пустые.** Пользователь планирует их заполнять, но шапка не содержит чётких инструкций «когда фиксировать», «как заполнять» и **«когда Claude должен сам предлагать запись»**.
6. **`CLAUDE_CODE_RULES.md` (9.5K) дублирует часть глобального `~/.claude/CLAUDE.md`** (правила «не навреди», тесты, шаги, отчёт). Полезные пункты (про паспорт, паттерны) — переедут в новый CLAUDE.md.
7. **Нет `.claude/settings.json`** — на каждое чтение/`ls`/`git status` всплывает промпт разрешения.

### Что хочется получить

- Чистый корень — только то, что отдаётся в браузер пользователю + 4 ключевых `.md` (`README`, `CLAUDE.md`, `PROJECT_PASSPORT.md`, `SPEC.md` как индекс).
- Журналы лежат в `docs/journal/` с понятной инструкцией по заполнению.
- SPEC нарезан по логическим разделам, корневой `SPEC.md` — оглавление-индекс.
- Все скиллы в `.claude/skills/<имя>/SKILL.md`.
- CLAUDE.md в новом чате достаточно, чтобы Claude знал, какой файл читать на каждый тип задачи, и не тратил токены на лишнее.
- Закреплены правила: русский язык в Bash-`description`, TodoWrite и AskUserQuestion; прогон каждого входящего промпта через скилл `uluchshatel-promptov`.

## 2. Решение по архитектуре

**Выбран подход B** («Структурированная разметка»). Альтернативы — A (минимальная чистка) и C (отдельная папка `ai/` для всего AI-контекста) — отвергнуты:

- A — оставляет журналы в корне, не решает проблему засорённости.
- C — ломает стандартное соглашение Claude Code (CLAUDE.md в корне репо), создаёт риски совместимости с другими подрядчиками.

## 4. Содержимое нового CLAUDE.md

Целевой объём ~8K. Структура:

1. **Шапка:** одна строка про назначение и приоритет.
2. **Старт сессии:** обязательно прочитать `PROJECT_PASSPORT.md` (статус и приоритеты), быстро глянуть `docs/journal/DECISIONS.md` (если есть записи).
3. **Карта проекта** — короткая таблица «когда что читать»:
   - `PROJECT_PASSPORT.md` — всегда в начале сессии.
   - `SPEC.md` (индекс) → `docs/spec/NN-*.md` — только нужный раздел по теме задачи.
   - `docs/quiz-spec.md` — только если правишь квиз.
   - `docs/journal/*` — перед/после сессии и при работе с багами.
   - `.claude/skills/*` — активируются автоматически по триггерам.
4. **Языковые табу** — текущий блок (Instagram, Meta, блокировка, центральная боль, вайбкодинг, технический жаргон, «дёшево», громкие обещания) — без изменений.
5. **Язык интерфейса (новое):** Bash-`description`, TodoWrite (`content`/`activeForm`), AskUserQuestion (вопросы и опции) — **всегда на русском**. Имена файлов, переменные, идентификаторы кода — английские.
6. **Прогон промптов через `uluchshatel-promptov` (новое):** каждый входящий **промпт-команда** пользователя сначала улучшается, потом исполняется. **Не прогоняем** ответы на вопросы Claude (выбор варианта, «да/нет», уточнения) и подтверждения действий.
7. **Журналы — когда читать и когда предлагать запись:**
   - `SESSIONS.md` — **читать в начале каждого нового чата** (последнюю запись для контекста «что только что делали»). В конце сессии — предлагать запись с тремя блоками: «Сделано», «Оставшиеся задачи», «Контекст для следующей сессии». **Сверху новые записи**, чтобы свежий контекст был в начале файла.
   - `DECISIONS.md` — читать в начале сессии, предлагать запись после архитектурного выбора («выбрали X вместо Y» / «отложили до Z»).
   - `PATTERNS.md` — читать при возврате к сложной задаче, предлагать запись после нестандартного обходного решения.
   - `CASES.md` — читать при работе с багами, предлагать запись при найденном баге или странном поведении.
   - Формат записей унаследован из текущих шаблонов; шаблон SESSIONS.md см. в §7 ниже.
8. **Что трогать в квизе / что не трогать** — текущий блок без изменений.
9. **Тяжёлые файлы — читать только по необходимости:**
   - `docs/superpowers/plans/2026-04-28-landing-implementation.md` (~100K) — только при работе над задачей из плана.
   - `docs/spec/04-design.md` — только при дизайн-работе.
   - `.claude/skills/152-fz/*` — активируются скилл-триггером, вручную не читать.
10. **Коммиты** — на русском (вынесено из паспорта/глобальной памяти, чтобы было под рукой).

### Что НЕ переносим из CLAUDE_CODE_RULES.md

Дубли глобального `~/.claude/CLAUDE.md` (не навреди, тесты, шаги, отчёт после выполнения, обратная совместимость, git-гигиена) — глобальные правила уже работают.

### Что переносим из CLAUDE_CODE_RULES.md

- Упоминание про паспорт проекта (но в нашей формулировке).
- Принцип «один коммит = одна логическая единица» — в раздел про коммиты.

## 6. Содержимое `.claude/settings.json`

Минимальный набор автодопусков для read-only команд. Записи добавляются через `permissions.allow`:

```json
{
  "permissions": {
    "allow": [
      "Bash(git status:*)",
      "Bash(git diff:*)",
      "Bash(git log:*)",
      "Bash(ls:*)",
      "Bash(du:*)",
      "Bash(wc:*)"
    ]
  }
}
```

Без `permissions.deny` — оставляем дефолтное поведение Claude Code на остальных командах.

## 7. Обновление шапок журналов и новый SESSIONS.md

### Шапки CASES/PATTERNS/DECISIONS

Каждый из этих файлов получает обновлённую шапку с тремя блоками:

1. **Когда фиксировать** (триггеры — баг, архитектурный выбор, обходное решение).
2. **Как заполнять за 2 минуты** (шаги).
3. **Шаблон записи** (текущий формат).

Дополнительно — комментарий для Claude: «при наступлении триггера предложи запись формулировкой `Зафиксировать в [файл]? [да/нет/правка]`».

### Новый SESSIONS.md

Создаётся с нуля как хронологический журнал сессий. Цель — дешёвый cold-start новых чатов: Claude читает `PROJECT_PASSPORT.md` (что сейчас) + последнюю запись в SESSIONS.md (что только что делали и что осталось).

Структура файла:

- **Шапка:** назначение и инструкция «новый чат читает паспорт + последнюю запись здесь».
- **Когда фиксировать:** в конце каждой содержательной сессии.
- **Когда Claude должен предлагать запись:** в конце сессии (по сигналам «всё», «закончили», «достаточно», или после серии коммитов и явной паузы). Формулировка: «Записать саммари сессии в SESSIONS.md? [да/нет/правка]».
- **Формат записи:**
  ```
  ## YYYY-MM-DD — [Заголовок сессии в 3–6 слов]

  **Сделано:**
  - [пункт]

  **Оставшиеся задачи:**
  - [пункт]

  **Контекст для следующей сессии:**
  - [что важно помнить, нерешённые вопросы, предполагаемый следующий шаг]
  ```
- **Порядок записей:** сверху новые, снизу старые — чтобы свежий контекст был в начале файла и его не нужно было искать.

## 10. Что НЕ входит в эту работу

- Оптимизация `photo.png` (1.8 MB) — отдельная задача.
- Создание хуков для автозапуска улучшателя промптов — пользователь хочет ручной прогон, скилл активируется триггером.
- Никакие правки HTML/CSS/JS — только организационная структура и Markdown.
- `.gitignore` — оставляем как есть.
- Обновление PROJECT_PASSPORT.md — он актуальный, только в разделе «Структура проекта» при следующем обновлении надо отразить новые пути (это делается в следующей сессии, не сейчас).

## 11. Риски и митигации

| Риск | Митигация |
|---|---|
| При нарезке SPEC.md теряется кусок текста. | Перед удалением старого `SPEC.md` сделать `git diff` и пройтись по разделам; убедиться, что `cat docs/spec/*.md` покрывает все `##`-заголовки оригинала. |
| Кто-то из подрядчиков ссылается на старые пути `CASES.md`/`PATTERNS.md` в корне. | Поиск по репо: `grep -r "CASES.md\|PATTERNS.md\|DECISIONS.md\|CLAUDE_CODE_RULES"`. Обновить все найденные ссылки. |
| Распаковка `uluchshatel-promptov.skill` (zip) — кодировка/пути. | Перед удалением zip проверить, что распакованный `SKILL.md` читается и валиден (содержит frontmatter с `name:` и `description:`). |
