# CASES.md — Журнал проблемных кейсов

> **Зачем:** фиксировать ситуации, когда система работает не так, как ожидалось.
> Каждый кейс — будущий тест и основание для задачи.
>
> **Нужен ли этот файл?** Да, если в проекте есть поведение, которое можно проверить
> на реальных данных (сканер, парсер, генератор, API). Нет, если проект — статичный
> сайт или утилита без сложной логики. В этом случае удали файл.

---

## Когда записывать кейс

- Результат работы системы не совпадает с ожиданием
- Исправлен баг — запиши со статусом «исправлено» для проверки регрессии
- Пользователь сообщил о проблеме

## Как записать за 2 минуты

1. Скопируй шаблон, вставь в конец раздела «Кейсы»
2. Заполни поля со звёздочкой ★ — остальное можно дописать позже
3. Номер — следующий по порядку

---

## Шаблон

```
### CASE-NNN: [суть проблемы в 5–7 словах]
- ★ **Дата:** ГГГГ-ММ-ДД
- ★ **Вход:** [что подали на вход — URL, файл, запрос...]
- ★ **Что произошло:** [что система выдала]
- ★ **Что ожидалось:** [что должно было быть]
- **Подробности:** [дополнительный контекст, если нужен]
- **Причина:** [если понятна, иначе не заполняй]
- **Статус:** открыт | исправлено в [задача] | не воспроизводится | won't fix (причина)
```

---

## Как использовать с Claude Code

```
Прочитай CASES.md. Вот открытые кейсы для исследования.
Начни с CASE-NNN: воспроизведи, найди причину, предложи исправление.
```
